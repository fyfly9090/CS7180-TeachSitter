# Reflection: TeachSitter

_Zixin Lin_

My role on TeachSitter was the parent-facing half of the product: the search page, the AI matching pipeline (`/api/match`), the booking flow, and the eval system that scores every match result. I also built the `/issue-plan` slash command skill that Yun and I both relied on throughout the project. Looking back, the biggest shift I experienced wasn't in the features themselves, but in how I think about what it means to "develop" software.

**Building a custom skill changed how I plan.** Early in the project I noticed a recurring pain point: I'd open a feature branch without a clear picture of which files needed to touch which tables, and I'd discover dependencies mid-implementation. I built `/issue-plan` to close that gap. The skill fetches the GitHub issue via the MCP server, launches parallel Explore agents targeting different angles of the codebase (data flow, existing validation patterns, test structure), then gates a Plan agent on explicit user approval before any code is written. When I iterated on the skill itself, I first added structured issue display after finding the raw API response was unreadable, then tuned the Explore prompts. The process taught me something I didn't expect: writing a skill is software engineering. It requires the same discipline as any other interface: clear inputs, predictable outputs, testable behavior. After building it, I stopped treating planning as overhead and started treating it as the first deliverable.

**The Writer/Reviewer pattern became real when I applied it to security.** For most features, I ran two Claude Code passes: the first generates implementation, the second reviews for edge cases and spec compliance. For `/api/evals`, I went a step further and invoked the `security-reviewer` agent, a specialized subagent that reviews endpoint design in C.L.E.A.R. format (Code quality, Logic errors, Expectations from spec, Accessibility gaps, Risks), before writing a single line of implementation. It caught a `select("*")` on `match_evals` that would have silently leaked future schema columns and flagged which auth pattern was consistent with the rest of the codebase. The fix was one line. What the review reinforced was a shift in habit: treat the design review as a first-class artifact rather than a post-hoc checkbox. Human review still runs after both passes, but by the time I touch the code, the obvious issues are already resolved.

**TDD stopped being a discipline and became a communication tool.** The 13 tests I wrote for `/api/match` before any implementation covered auth, Zod validation, the `Promise.any()` race between Gemini and Claude, and eval logging. They turned out to be the most useful documentation I produced. Two weeks later, when I needed to add the booking-modification path, those tests told me exactly what the route was supposed to do without reading a line of implementation. The RED commit, the one where all 13 tests fail and I commit them anyway, is the one I now think of as the real spec.

**What I underestimated was context management.** CLAUDE.md isn't just a checklist; it's a living contract between the developer and every future agent session. When I added `encodeURIComponent` as an explicit requirement after CodeQL flagged a missing call on the `name` query parameter, I updated CLAUDE.md the same day. That update caught the same class of bug in two subsequent routes without any human intervention. Keeping the document current, not as passive documentation but as active instructions, is the highest-leverage habit I'm taking out of this project.

The model that emerged over four weeks: I design the contract (tests, CLAUDE.md, issue plan), AI produces the implementation, I verify behavior. That division of labor is more productive than I expected, and it's the one I'll carry forward.
