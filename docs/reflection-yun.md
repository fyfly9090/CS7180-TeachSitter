# Reflection — TeachSitter

_Yun Feng_

Looking back on the TeachSitter project, I was surprised that the biggest challenge wasn't just writing the code itself—it was building the safety nets that allowed me to work quickly without breaking the app. I was responsible for the teacher-facing side of the platform, as well as the underlying security, error tracking, automated testing, and our AI integrations. In hindsight, these weren't just separate tasks; they all shared a common goal: building a system we could trust.

My biggest takeaway was that having multiple layers of security isn't just being overly cautious; it actually makes the system smoother and easier to work with. For example, when building the feature to update teacher profiles, I almost relied solely on the database to block unauthorized changes. But I decided to add an extra security check in the application logic too. Within a week, that extra layer saved me twice—once when a database update briefly failed, and again when I needed to show users a clear "access denied" message instead of a confusing technical error. Having two layers isn't a waste of time; each serves a specific purpose. Getting that security sequence right from the start made adding every new feature much easier.

In terms of workflow, I learned how valuable it is to use tools that let me work on multiple parts of the project at the same time. During our second sprint, I needed to run the app to build a new feature while simultaneously writing tests for it in a separate workspace. Finding a way to have both open at once without constantly switching back and forth was a game-changer. It’s a small adjustment, but it completely changed how I organize my tasks, allowing me to build features and test them side-by-side.

Learning to work alongside an AI assistant like Claude was a unique challenge. At first, I just asked it to write code for me, but the results—while functional—were messy and didn't match the rest of our project's style. Everything changed when I realized that giving the AI highly specific instructions, clear examples, and strict rules produced much better results than vague requests. I eventually set up automated checks to catch my own mistakes and keep the AI from going off track. By the end of the project, I was spending most of my time planning the logic and reviewing the work, rather than just typing.

Looking back on what I would change: I finished the booking feature in a way that works perfectly on the user interface—it automatically hides times that are already booked. However, I didn't add a final check in the underlying system to prevent overlapping appointments. While a regular user wouldn't see an issue, someone with technical knowledge could potentially force a double-booking by bypassing the app's interface. Fixing this the right way is something I put off, but shouldn't have. I need to stop telling myself, "It's fine, the app catches the error somewhere else."

The other thing I would do differently is coordinate earlier with Zixin on exactly how our data should be structured. We each built half of the product separately, which was incredibly fast, but when we finally connected our pieces, we found a few minor mismatches. A quick 15-minute conversation during our first week could have prevented those headaches. For my next project, I’ll make sure we spend an hour right at the beginning to agree on exactly how our systems will communicate before anyone starts building.
